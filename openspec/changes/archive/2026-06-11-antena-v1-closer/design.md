# Design: antena-v1-closer

## Context

Antena was a working Reddit/X.com-style news feed (committed in `feature/antena-top-tier` and HEAD's `App.tsx` 512-line version). A WebLLM agent experiment replaced it in the working tree, breaking the production feed. The user wants to:

1. Restore the feed as the canonical frontend
2. Polish for mobile-first excellence
3. Use Cloudflare 100% (every service available on the platform)
4. Close the design (no more iteration)

The full design rationale is in `docs/superpowers/specs/2026-06-11-antena-v1-closer-design.md`. This design.md focuses on technical decisions for OpenSpec.

## Goals / Non-Goals

**Goals:**
- Mobile-first Reddit/X.com feed (top tabs + bottom nav)
- Cloudflare 100% (Pages, Workers, D1, KV, R2, Image Resizing, Cache API, Vectorize, Analytics Engine, Queues, Cron Triggers, Pages Functions, Turnstile)
- Best tech for each layer: Tailwind 4, Drizzle ORM, Solid Query, Zod
- Fix all 9 P0/P1 bugs
- Delete all 18 community-service stubs
- WebLLM agent moved to `feature/webllm-agent` branch (archived)
- Performance budget: LCP < 1.8s, JS shell < 80KB, Lighthouse > 90
- Operational "cerrado" checklist (in superpowers spec)

**Non-Goals:**
- No new features beyond what's in the superpowers spec
- No architecture rewrites beyond Cloudflare-native
- No AKIRA changes (Python pipeline is the source of truth)
- No accounts/auth/comments/reactions real (deferred v2)
- No community services, push notifications, i18n, RSS reader, newsletter

## Decisions

### Decision 1: Astro 5 + Solid.js stays

Astro's islands architecture with `output: "static"` is the best fit for a mobile-first feed:
- Pre-rendered HTML at the edge (no SSR cost, no hydration of unused components)
- Solid.js islands hydrate only what's interactive (cards, nav, tabs)
- `client:visible` for below-the-fold, `client:idle` for non-critical

Alternatives considered:
- Next.js — too heavy, hydration cost, RSC doesn't help here
- SvelteKit — fine but we already have Solid components
- Solid Start — overkill, no static output advantage

### Decision 2: Hono stays

Hono 4.7 is the de facto standard for Cloudflare Workers:
- ~12KB runtime
- Type-safe with `Hono<{Bindings: Env}>`
- Middleware ecosystem (cors, logger, cache)
- `c.executionCtx.waitUntil()` and `c.env` bindings work out of the box

Alternatives considered:
- itty-router — smaller but less ergonomic
- Raw fetch handlers — too much boilerplate

### Decision 3: Tailwind 3 → Tailwind 4

Migrate to Tailwind 4 with `@tailwindcss/vite`:
- Bundle ~50% smaller
- CSS-first config (`@theme` directive in CSS)
- Better DX, faster builds
- Compatible with Astro 5 via Vite plugin (drop `@astrojs/tailwind`)

Migration steps:
- Install `tailwindcss@4` and `@tailwindcss/vite`
- Remove `@astrojs/tailwind`
- Convert `tailwind.config.js` to CSS `@theme` block in `design-tokens.css`
- Update `astro.config.mjs` to use `vite.plugins: [tailwindcss()]` instead of `integrations: [tailwind()]`

### Decision 4: Drizzle ORM for D1

Replace raw SQL with Drizzle for type-safe D1 queries:
- Type-safe queries, autocompletado en TS
- Migrations via `drizzle-kit` (no manual SQL files)
- Works in Workers (no Node deps)
- Better DX than `better-sqlite3` (which is the wrong tool for prod)

Keep D1 migrations version-controlled. Drizzle generates them from schema.

### Decision 5: Solid Query for data fetching

Add `@tanstack/solid-query@5`:
- Cache + dedup + invalidation
- Optimistic updates (bookmarks)
- Background refetch on focus
- Works perfectly with `createResource` from Solid

Alternative considered: native `createResource` — works but you reinvent cache/dedup.

### Decision 6: Zod for validation

Add `zod@3` for:
- API request params validation
- Form input validation
- Type guards (inferred TS types from schemas)

Single source of truth for shapes.

### Decision 7: WebLLM moved to branch

The WebLLM experiment doesn't fit v1 because:
- Not edge-native (WebLLM is a 1.2GB browser download)
- Not mobile-first (blocks UI for minutes on first load)
- Not Cloudflare (runs in browser, not on platform)

Move to `feature/webllm-agent` branch. The code is preserved in git history for future v2/v3 work (maybe with Workers AI instead of WebLLM).

### Decision 8: R2 + Image Resizing for images

Don't hotlink source images. R2 + Image Resizing:
- Worker endpoint `GET /img/:hash?w=400&fmt=avif&fit=cover`
- If R2 has it: return transformed
- If not: fetch from source, upload to R2, transform, cache 1 year
- AVIF preferred (saves 70% bandwidth), WebP fallback, jpg last

`<img>` tag uses srcset generated by `lib/image.ts`:
```html
<img srcset="/img/abc123?w=400 400w, /img/abc123?w=800 800w" sizes="...">
```

### Decision 9: Cache API at the edge

Every public endpoint sets Cache-Control and stores in `caches.default`:
- Feed: 60s TTL + 5min SWR
- Article: 5min TTL + 1h SWR
- Cluster: 5min TTL
- User-specific (bookmarks): no cache
- Search: no cache (Vectorize direct)

Cache keys are deterministic: `feed:public:${cat}:${loc}:${sort}:${page}`.

### Decision 10: Vectorize for semantic search

Index `news_embeddings` (384-dim, cosine) with title+summary embeddings:
- "Para vos" uses vector similarity to diversify
- Search combines FTS5 (lexical) + Vectorize (semantic) for best results
- Cron trigger refreshes index every 2h

Alternative: pure FTS5 — faster but no semantic understanding.

### Decision 11: Analytics Engine for read events

No cookies, no PII, edge-processed:
- Each card view + dwell time + scroll depth → writeDataPoint
- Dataset `feed_events`
- Powers "Para vos" personalization in v2 (without auth)

Alternative: Google Analytics — privacy concerns, JS overhead.

### Decision 12: Pages Functions for SSR

Article pages (`/noticia/[id]`) use Pages Functions for:
- Server-side render with OG tags (SEO + share previews)
- Cache at edge, serve as static after first request
- Better than client-side fetch (one round-trip less)

Home, bookmarks, search stay as static Astro pages (no SSR needed).

### Decision 13: Touch targets and haptics

All interactive elements ≥ 44pt (Apple HIG):
- Card tap target: full card row
- Action buttons: 44x44pt min
- Bottom nav: 56pt height + safe area
- Top tabs: 44pt height

Haptic feedback sparingly (see `design-tokens.css` haptic table):
- selection: tab change, card tap
- impactLight: vote
- impactMedium: pull-to-refresh trigger
- notificationSuccess: bookmark, share

No haptic on scroll, no on every render.

### Decision 14: Skeleton loading, no spinners

Every async load shows a skeleton (not a spinner):
- Skeleton variants per content type (card, cluster, hero)
- Shimmer animation (linear gradient, 1.5s loop)
- Skeletons disappear on first paint of real content (no flash)

Spinners only for explicit user-triggered refresh (pull-to-refresh).

## Architecture (technical)

### Data flow

```
[Astro static] index.astro → App.tsx (client:only="solid-js")
  → onMount: TanStack Query client created, haptics initialized
  → createResource(fetchFeed) → first paint
  → IntersectionObserver sentinel → append more items
  → on click → setSelectedNews + updateURL({view: 'article', id})

[API Worker] /api/news/feed
  → Cache.match(cacheKey) → if hit, return
  → D1 SELECT (with bias_score, category, location)
  → Drizzle query
  → response.headers.set('Cache-Control', 'public, max-age=60, stale-while-revalidate=300')
  → Cache.put(cacheKey, response.clone())
  → return

[API Worker] /img/:hash
  → R2.head(hash) → if hit, Image Resizing transform, return
  → D1 SELECT source_url by hash
  → fetch(source_url) → R2.put(hash, blob) async via queue
  → Image Resizing transform → return
  → Cache-Control: public, max-age=31536000, immutable

[Pages Function] /noticia/[id]
  → D1 SELECT news + cluster
  → render Astro template with OG meta tags
  → Cache at edge
  → return HTML

[Queue consumer] image-pipeline
  → receive { hash, source_url }
  → fetch(source_url) → R2.put(hash, blob)
  → write Analytics Engine event

[Cron trigger] 0 */2 * * *
  → refresh Vectorize index from D1 (new/changed news)
  → clean stale cache entries
  → emit health metrics to Analytics Engine
```

### State management

App-level signals (Solid):
- `currentView: Signal<'feed' | 'article' | 'search' | 'bookmarks' | 'menu'>`
- `activeTab: Signal<'para-vos' | 'siguiendo' | 'explorar'>`
- `activeCategory: Signal<string | null>`
- `activeLocation: Signal<number | null>`
- `sort: Signal<'hot' | 'new' | 'top' | 'controversial'>`
- `feed: Resource<{ items, hasMore }>` (TanStack Query backed)
- `selectedNews: Signal<NewsItem | null>`
- `urlState: Store` (synced with `?cat=&loc=&view=&id=` via `lib/urlState.ts`)

Per-component local state:
- `useHaptic()` returns `{ vibrate: (kind) => void }`
- `useBookmarks()` returns `{ bookmarks, isBookmarked, toggle }` (IndexedDB + Solid Query)
- `useFeedState()` returns `{ items, fetchNext, hasMore, isFetching }` (TanStack Query infinite)

### Cache layers (defense in depth)

1. **Browser HTTP cache** (Service Worker, 5min for feed, 1h for article)
2. **Cloudflare Cache API** (60s TTL, 5min SWR, edge)
3. **D1** (SQLite, persistent)
4. **KV** (key-value, fast reads, used for hot keys)

IndexedDB is client-side only (bookmarks, reading history, prefs).

### Routing

- `/` — home (feed)
- `/bookmarks` — bookmarks list
- `/search?q=` — search results
- `/noticia/[id]` — article (Pages Function SSR for SEO)
- `/categoria/[slug]` — category feed
- `/fuente/[id]` — source feed

All share the same App.tsx shell, no SPA router needed (Astro's file-based routing).

### Error handling

- API errors → toast with retry button, fall back to stale cache
- Image errors → fallback placeholder (SVG of source logo)
- Network offline → PWA service worker serves cached shell
- Zod validation errors → 400 with field-level details

### Security

- Cloudflare Turnstile on share/bookmark forms (bot protection)
- Zod validation on all API inputs
- CSP headers (no inline scripts, strict)
- Rate limiting via Cache API + IP hash (no auth = per-IP)
- No PII collected (Analytics Engine uses anonymous index)

## Risks / Trade-offs

- **[Risk] Tailwind 4 migration breaks existing classes**
  - Mitigation: most classes are stable; only `@apply` directives and config syntax change. Migration is mechanical.

- **[Risk] Drizzle adds runtime overhead vs raw SQL**
  - Mitigation: Drizzle is thin (no extra queries, just typed builders). Acceptable for type safety gain.

- **[Risk] Vectorize cold start on first search**
  - Mitigation: Cron trigger pre-warms index every 2h. First search might be slow in dev, fine in prod.

- **[Risk] R2 egress costs if image pipeline is popular**
  - Mitigation: Image Resizing + AVIF reduces bandwidth 70% vs raw. Set budget alerts in CF dashboard.

- **[Risk] Solid Query bundle size**
  - Mitigation: 13KB gzipped, within budget. Use selective imports.

- **[Risk] WebLLM branch rots**
  - Mitigation: tag commit before archiving, document in `docs/archived/webllm-experiment.md`.

- **[Risk] Mobile device fragmentation (Android 8-15)**
  - Mitigation: Test in mobilerun on real device, baseline Android 10. Document iOS Safari quirks.

- **[Risk] iOS Safari PWA install friction**
  - Mitigation: clear "Add to Home Screen" instructions, fallback to in-browser experience.

- **[Risk] TypeScript strict mode shows pre-existing errors in feed code**
  - Mitigation: fix during implementation (part of "TypeScript strict sin errores" in Cerrado).

## Migration Order

The tasks.md is structured in 8 phases that can be parallelized where safe:

1. **Foundation** — package.json updates, deps install, configs (Tailwind 4, Astro, Vitest)
2. **Data layer** — Drizzle schema, migrations, types
3. **API refactor** — Zod validation, Cache API on all routes, new routes (image, search, track)
4. **Frontend refactor** — revert App.tsx, refine components for mobile, fix bugs, delete stubs
5. **New components** — Skeleton, BottomSheet, EmptyState
6. **Cloudflare integration** — R2, Vectorize, Analytics, Queues, Cron, Pages Functions
7. **Testing** — unit, components, API, E2E with mobilerun
8. **Deploy & docs** — CI/CD, wrangler configs, AGENTS.md, README, docs

Each phase has a verification gate (tests pass, lighthouse, typecheck).

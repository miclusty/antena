# @akira/api

Cloudflare Worker API for Antena/AKIRA. Hono-based, deployed to Workers, backed by D1 (SQLite at the edge).

## Endpoints

See `src/routes/*` and the OpenSpec design doc at `docs/superpowers/specs/2026-06-11-antena-v1-closer-design.md`.

## Cloudflare bindings (declared in `wrangler.toml`)

| Binding | Type | Purpose |
|---------|------|---------|
| `DB` | D1 Database | News cards, sources, locations, categories, clusters, master articles |
| `CACHE` | KV Namespace | Edge cache for hot reads |
| `IMAGES` | R2 Bucket | Resized image cache |
| `VECTORS` | Vectorize Index | Semantic news search |
| `ANALYTICS` | Analytics Engine | Read events, dwell time, scroll depth |
| `IMAGE_QUEUE` | Queue | Async image processing pipeline |

Typed bindings live in `src/lib/types.ts` (Worker env) and `packages/antena/src/lib/cloudflare.ts` (Pages Functions env).

## Database migrations

Migrations are versioned SQL files in `migrations/`. They are auto-generated by `drizzle-kit` from `src/db/schema.ts`.

### Generate a new migration

```bash
# From packages/api
pnpm exec drizzle-kit generate
```

> **Never edit migration SQL by hand.** Modify `src/db/schema.ts` and regenerate. The journal (`migrations/meta/_journal.json`) tracks applied migrations and their diff snapshots.

### Apply locally (against miniflare + local D1)

```bash
wrangler d1 migrations apply DB --local
```

This runs every migration in `migrations/` that has not yet been applied to the local D1 database (`.wrangler/state/v3/d1/`).

### Apply to production (remote D1)

```bash
wrangler d1 migrations apply DB --remote
```

Use the `--env` flag when wrangler.toml defines multiple environments (e.g. `--env=production`).

### Required env vars

`drizzle-kit` and `wrangler` need the following for remote operations (set in `packages/api/.dev.vars` for local dev, and as Cloudflare secrets in CI for prod):

- `CLOUDFLARE_ACCOUNT_ID`
- `CLOUDFLARE_D1_DATABASE_ID`
- `CLOUDFLARE_API_TOKEN`

### Initial migration

`migrations/0000_initial_schema.sql` creates the six public-facing tables:

- `categories`
- `locations`
- `sources`
- `clusters`
- `news_cards`
- `master_articles`

It does **not** seed data — categories and locations are seeded by the AKIRA pipeline on first run (see `../../migrations/0001_complete_schema.sql` at the repo root for the canonical seed data).

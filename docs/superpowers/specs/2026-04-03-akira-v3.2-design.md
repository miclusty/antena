## Phase C: Auto-Healing Sources

### Architecture

```
SourceRecovery Service
├── attempt_recovery(source_url) → RecoveryResult
├── scan_failed_sources() → List[RecoveryResult]
├── get_failed_sources() → List[Source]
└── get_recovery_stats() → RecoveryStats
```

### Recovery Process

```
Source fails 5+ times
    ↓
Circuit breaker opens (is_circuit_open = 1)
    ↓
Recovery service detects (every 6 hours)
    ↓
Try RSS paths: /feed/, /rss, /feed/rss2, /rss2.xml
    ↓ (if not found)
Try WP API: /wp-json/wp/v2/posts?per_page=1
    ↓ (if not found)
Try Sitemap: /sitemap.xml, /sitemap_index.xml
    ↓ (if found)
Update source record:
  - Set new rss_url/wp_api_url/sitemap_url
  - Reset error_count = 0
  - Reset is_circuit_open = 0
  - Set is_active = 1
    ↓ (if not found)
Mark as permanently dead:
  - is_active = 0
  - deactivation_reason = "recovery_failed"
```

### New Endpoints

```
POST /admin/recover-source
{"url": "https://example.com"}

GET /admin/failed-sources
→ {"sources": [{"url": "...", "error_count": 8, "last_success": "..."}]}

GET /admin/recovery-stats
→ {"total_failed": 12, "recovered": 5, "permanently_dead": 3}
```

## Phase D: Smart Method Learning

### Current State (v3.1)

```python
# Only stores:
last_success_method = "rss"  # Last successful method
success_count = {"rss": 5}   # Count per method
consecutive_failures = 0     # Circuit breaker
```

**Limitations:**
- No speed consideration (RSS=3s vs WP=2s treated same)
- No time-based patterns
- Cascade order based on static priority, not performance

### Enhanced Design (v3.2)

#### 1. Weighted Scoring

```python
class MethodScore:
    """Score = (success_rate * 0.6) + (speed_score * 0.4)"""
    
    success_rate = success_count / (success_count + failure_count)
    avg_duration = sum(durations) / len(durations)
    speed_score = max(0, 1 - (avg_duration / 60000))  # 60s = 0 score
    
    score = (success_rate * 0.6) + (speed_score * 0.4)
```

**Example scores:**
| Method | Success Rate | Avg Duration | Speed Score | Final Score |
|--------|-------------|--------------|-------------|-------------|
| WP API | 95% | 2000ms | 0.97 | 0.96 |
| RSS | 90% | 3000ms | 0.95 | 0.95 |
| Newspaper | 70% | 30000ms | 0.50 | 0.62 |
| Playwright | 60% | 45000ms | 0.25 | 0.46 |

#### 2. Time-Based Patterns

```python
# Track method performance by hour
CREATE TABLE method_history (
    url TEXT,
    method TEXT,
    duration_ms INTEGER,
    success INTEGER,
    hour_of_day INTEGER,  -- 0-23
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
);

# Query: best method for this URL at current hour
SELECT method, AVG(duration_ms) as avg_duration, 
       SUM(success) * 1.0 / COUNT(*) as success_rate
FROM method_history
WHERE url = ? AND hour_of_day = ?
GROUP BY method
ORDER BY success_rate DESC, avg_duration ASC
LIMIT 1
```

#### 3. Cascade Optimization

```python
def get_optimized_cascade(self, url: str) -> List[Extractor]:
    """
    Build cascade order based on weighted scores.
    
    Old: RSS(100) → WP(90) → Newspaper(70) → ...
    New: WP(0.96) → RSS(0.95) → Newspaper(0.62) → ...
    """
    scores = self.get_method_scores(url)
    sorted_methods = sorted(scores, key=lambda x: x.score, reverse=True)
    
    return [self.get_extractor(m.method) for m in sorted_methods]
```
